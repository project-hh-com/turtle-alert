# 구현 계획: "앱 열어줘" 기능

## 기능 정의

트레이 메뉴 최상단에 **"앱 열어줘"** 항목 추가.
클릭 시 저장된 간격(intervalMin)으로 타이머를 즉시 시작하고, 첫 알림을 보내 앱이 활성화되었음을 사용자에게 알림.
이미 실행 중이면 해당 메뉴 항목을 비활성화(또는 숨김) 처리.

## 변경 파일

| 파일 | 변경 내용 |
|---|---|
| `lib.js` | `updateTrayMenu()` 내 메뉴 템플릿 최상단에 "앱 열어줘" 항목 추가 |
| `__tests__/lib.test.js` | "앱 열어줘" 클릭 시 타이머 시작 + 알림 발송 검증 테스트 추가 |

## 구현 상세

### 1. 메뉴 항목 추가 (`lib.js` — `updateTrayMenu()`)

```js
{
  label: "🐢 앱 열어줘!",
  enabled: !isRunning,
  click: () => {
    startTimer(intervalMin);
    sendAlert();  // 첫 알림으로 활성화 확인
  },
},
```

- 위치: 기존 "대기 중/실행 중" 상태 표시 바로 아래, separator 전
- `isRunning`이 true이면 `enabled: false`로 비활성화

### 2. 테스트 추가

- "앱 열어줘" 클릭 → `startTimer` 호출 확인
- "앱 열어줘" 클릭 → `sendAlert` 호출 확인  
- 이미 실행 중일 때 → 메뉴 항목 `enabled: false` 확인

## 핵심 위험 요소 (Red-Team 검토)

### 위험 1: 첫 알림이 타이머 카운트에 중복 집계됨

**위협**: `sendAlert()`가 `alertCount`를 증가시키므로, "앱 열어줘" 클릭 시 실제 스트레칭 알림이 아닌 "시작 알림"도 카운트에 포함되어 통계가 부풀려짐.

**방어책**: 시작 알림을 `sendAlert()` 대신 별도의 환영 Notification으로 분리. `alertCount` 증가 없이 "거북이경보 시작! 자세 똑바로 하세요~" 메시지만 발송.

### 위험 2: 빠른 연타로 타이머 다중 시작

**위협**: Electron 메뉴 클릭 이벤트가 연속 발생할 경우 `startTimer()`가 여러 번 호출될 수 있음. 현재 `startTimer` 내부에서 `stopTimer()`를 먼저 호출하므로 interval 누수는 없지만, `sendAlert()`가 여러 번 호출되어 알림 폭탄이 될 수 있음.

**방어책**: `startTimer()` 내부의 `stopTimer()` 호출이 이미 방어 역할을 함. 추가로, 클릭 핸들러에서 `isRunning` 체크를 한번 더 수행하거나, 메뉴 `enabled: !isRunning` 조건으로 UI 레벨에서 차단 (이미 계획에 포함).

### 위험 3: 앱 컨셉 훼손 — "앱 열어줘"가 BrowserWindow를 연상시킴

**위협**: "앱 열어줘"라는 문구가 사용자에게 별도 창이 열릴 것이라는 기대를 줄 수 있음. 이 앱은 창 없는 메뉴바 앱이므로 컨셉 혼란 유발.

**방어책**: 레이블을 **"🐢 거북이 깨워!"** 또는 **"🐢 감시 시작!"** 등으로 변경하여 "타이머 활성화" 의미를 명확히 전달. 창이 열리는 것이 아님을 톤앤매너로 암시.

## 최종 권장사항

1. 레이블은 `"🐢 감시 시작!"` 권장 (앱 컨셉과 일치)
2. 시작 시 환영 알림은 `alertCount` 증가 없이 별도 처리
3. 변경 범위가 `updateTrayMenu()` 한 함수 내로 제한되어 영향도 낮음
